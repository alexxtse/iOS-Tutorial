这界面是我的还是你的？

这界面是我的还是你的？

原文地址 / 作者：John Saito

翻译于：2017/09/18，Alex Tse

「我的电脑」的演变

都还记得以前 Windows 那个「我的电脑」的图标吗？这是个光荣的小图标，它代表了你电脑上所有东西 - 你的程序，你的工作，你的数字作品等。

在 Windows 后面的版本中，微软把图标的名字改为「计算机」，后来又改为「此电脑」(This PC)。他们把名字改了，难道是因为「我的」具有误导性？不一致性？还是说这是多余的呢？

属于你的东西要怎么称呼？

观察下你使用的 App，你会发现用于表示你的东西的界面是没有标准的方法可参考的。有的说这是我的东西，而有些说这是你的东西。

假如你正在设计界面，这个词是应该从用户的角度还是从产品的角度来决定？我认为这是一个很微妙的差异，取决于你希望用户在使用你的产品时的感觉如何。

从「我的」角度考虑

在界面上使用「我的」意味着产品是用户的延伸，就好像产品代表着用户标记的东西。「我的」感觉是个人的，这种感觉就像是你可以去定义它和控制它。

通过这个逻辑，当需要强调隐私，个性化，所有权时，「我的」可能会更适合。这或许就是几年前使用「我的电脑」的原因，因为当时电脑几乎是单机体验，用户通常没有共享文件，所有的东西都在那个小图标里面安全的待着。

从「你的」角度考虑

在界面上使用「你的」意味着产品正在和你说话，产品就像是你的个人助理，帮助你完成某些事情。『这里是你的音乐。这里是你的订单。』

根据这个逻辑，当你希望你的产品听起来像是在对话，像是在帮助用户完成某项任务，使用「你的」可能会更合适。无论是支付账单，预约，填写税单，能帮助用户更快，更智能，更简单的完成某项事情。

如今的电脑和 App 几乎是扮演着一个私人助理的角色。像 Siri，Alexa 和 Cortana，她们帮你做笔记，提醒你买牛奶，大声的帮你读邮件。

许多不同的 App，包括 Medium，都会给你推荐一些东西。在我看来它是一个推荐故事给我阅读的私人助手，而且我认为这种趋势以后会更加的普遍，我们以后可能会看到越来越多的 App 使用的是「你的」而不是「我的」。

考虑删去

和设计中大多数的事情一样，没有一种解决方案是可以符合所有情况的。但现在有很多产品只要是属于用户的东西，就会把「我的」或者「你的」这类词删掉。

这种去掉「我的」的方式或许和当初 Windows 把「我的电脑」(My Computer)改成「计算机」(Computer)一样吧。

不幸的是，去掉「我的」或者「你的」不一定是最优办法，而且还有可能产生更多歧义。有时你真的需要区分这到底是谁的东西，例如：在 YouTube 上，你不能直接叫「频道」(Channel)，因为这不知道指的是你订阅的频道，还是 YouTube 给你推荐的频道。

这或许（只是或许）是 Windows 最后把「计算机」(Computer)改成「此电脑」(This PC)的原因吧。因为「计算机」本身定义太模拟两可，所以他们需要明确的指出是这台电脑。

综合考虑(Putting it all into perspective)

到目前为止，我主要讨论的是在界面上展示属于你的东西，这只是为用户设计所遇到的一小部分问题，接下来还有许多需要考虑的：按钮上的文字写什么？怎么说明？需要放到设置里面吗？等等。

有许多不同的方法可以解决这类问题，但下面是我喜欢遵循的规则：

  • 使用「我」的情况：在用户和产品交互的时候，使用我、我的、属于我的。例如点击按钮或点按复选框等。但只有在你必须要明确说明的情况下才能添加这些词。
  • 使用「你」的情况：在产品向用户提问、提供说明、描述某些事情的时候，使用你、你的。可以想象下私人助手会对你说什么。

从「我们的」角度考虑

在文章结束之前，我还想提一个常见的：「我们的」。即在界面上使用「我们」，「我们的」。

使用「我们」，「我们的」实际上是增加了第三个参与者 - 产品背后的人。这是表明有真的人在开发这些产品，而不是只有一些冷冰冰的程序机器。

如果你的产品是提供人类生产力服务(people-powered services)的，如烹饪，设计，清洁等，添加「我们」就更有人情味了。『在这里我们能帮到你。』，『看看我们的服务。』，这样可以让用户知道真正的人就在这些死板窗口和提示框的后面等着，让用户得以放松。

另一方面，如果你的产品是像 Google 搜索引擎这样的自动化工具，「我们」这个词就会感到误导，因为没有真的人来处理你的搜索。事实上，Google 编写 的 UI 指南里面，建议不要在界面中对大多数的事情使用「我们」。

你的考虑是什么？

我写这篇文章是因为这个问题总是一次又一次的出现在设计师、开发者和文案编辑者身上。为什么要在这里使用「我的」？为什么要在那里使用「你的」？然而，我很少看到有类似的指南。

对于这方面的设计思考，你有自己的指南吗？如果有，我很想听听你是怎么考虑的。
